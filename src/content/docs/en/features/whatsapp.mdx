The WhatsApp channel lets users send messages to ThairaAI through WhatsApp and receive AI replies back — using the same AI backends available in the desktop app.

## Architecture

ThairaAI connects to WhatsApp through a separate **whatsapp-api** server (based on [CodeChat](https://github.com/code-chat-br/whatsapp-api) / Baileys). ThairaAI never connects to WhatsApp directly.

```
WhatsApp ↔ whatsapp-api (:8084) ↔ ThairaAI (:25809)
```

- **Inbound**: whatsapp-api forwards received messages to ThairaAI via webhook
- **Outbound**: ThairaAI calls whatsapp-api's REST API to send replies

## Prerequisites

1. A running **whatsapp-api** instance (default port 8084)
2. A WhatsApp account connected to that instance
3. ThairaAI desktop app

## Configuration

Go to **Settings → Channels → WhatsApp** and fill in:

| Field | Description | Default |
|---|---|---|
| Server URL | Base URL of your whatsapp-api server | `http://localhost:8084` |
| Instance Name | Name of the WhatsApp instance in whatsapp-api | `thairaai` |
| Global API Key | `AUTHENTICATION_GLOBAL_AUTH_TOKEN` from whatsapp-api's `.env` | auto-discovered |

<Aside type="tip">
  If whatsapp-api lives in a sibling directory to ThairaAI, the API key is discovered automatically from its `.env` file — no manual entry needed.
</Aside>

Click **Save & Enable**. ThairaAI will:
1. Auto-start its internal HTTP server (if not already running for WebUI)
2. Create or resume the whatsapp-api instance
3. Register the webhook URL with whatsapp-api automatically

## User pairing

WhatsApp users must be approved before the AI responds to them.

1. Send any message to the WhatsApp number
2. ThairaAI replies with a 6-digit pairing code
3. Open **Settings → Channels → WhatsApp → Pending Pairings** in the desktop app
4. Click **Approve**

After approval, all messages from that number receive AI replies.

## Supported message types

| Type | How it's handled |
|---|---|
| Text | Forwarded to the AI as-is |
| Image | Downloaded to a temp file; the AI receives the local path and any caption as text |
| Voice message | Converted from OGG/Opus to WAV, transcribed via Speech-to-Text, then forwarded as text |
| Audio file | Same pipeline as voice messages |

<Aside type="tip">
  Voice and audio transcription require Speech-to-Text to be enabled. Go to **Settings → Tools → Speech to Text** and choose a provider. The **Local Whisper** option runs entirely on your machine with no API key required.
</Aside>

## Conversation behavior

- **Replies**: ThairaAI sends AI replies as new WhatsApp messages (WhatsApp does not support message editing)
- **Thinking indicator**: A `⏳ Thinking...` message is sent while the AI processes; the final reply follows as a separate message
- **History**: Conversation context is preserved within a session. If ThairaAI restarts or the session is idle for more than 5 minutes, the previous messages are automatically injected as context into the next request
- **Plain text only**: WhatsApp renders plain text — HTML formatting is stripped automatically

## Selecting an AI backend

In **Settings → Channels → WhatsApp**, choose the AI agent and model used for WhatsApp conversations. This is independent of your desktop chat configuration.

## Disabling

Toggle the channel off in Settings. ThairaAI will deregister the webhook from whatsapp-api automatically.

## WhatsApp Web tab

When you open a WhatsApp conversation in the desktop app, the conversation area shows two tabs:

| Tab | Content |
|---|---|
| **Chat** | The AI assistant chat — messages sent and received through the WhatsApp channel |
| **WhatsApp Web** | An embedded WhatsApp Web browser (`web.whatsapp.com`) |

The WhatsApp Web view runs as a **global singleton**: the embedded browser is never destroyed when you switch between conversations or navigate elsewhere in the app. Scan the QR code once — the session persists for the life of the app window (and across restarts, since the browser storage is isolated under `persist:whatsapp-web`).

<Aside type="tip">
  Switching between WhatsApp conversations keeps the WhatsApp Web tab open and already logged in. No re-scan is needed.
</Aside>

## Business Mode <Badge text="Enterprise" variant="caution" />

Business Mode flips the paradigm: instead of you chatting with your own assistant, your **clients** message your WhatsApp number and the AI responds on your behalf automatically — no manual pairing required.

This is ideal for professionals who handle client inquiries through WhatsApp: therapists, law firms, clinics, consultancies, and any business that receives contacts through the app.

<Aside type="caution">
  Business Mode is an **Enterprise** feature. Anyone who messages the configured number will receive automatic AI replies without prior approval. Configure carefully.
</Aside>

### How to enable

**Option 1 — Home screen**: click the toggle pill in the quick-action bar. The WhatsApp icon turns green when active. When enabled, you receive a confirmation message on your own WhatsApp number.

**Option 2 — Settings**: go to **Settings → Channels → WhatsApp → Business Mode** and toggle it on.

### Configuration fields

| Field | Description | Example |
|---|---|---|
| Business name | Identifies your practice to the AI | `Silva Psychology Clinic` |
| Professional name | Name the AI uses when referring to you | `Dr. Ana Silva` |
| Working hours | Communicated to clients asking about availability | `Mon–Fri, 9am–6pm` |
| Welcome message | First message sent to new contacts | *(auto-generated if left empty)* |

### What changes with Business Mode on

| Behavior | Standard mode | Business Mode |
|---|---|---|
| New contact | Receives pairing code | Receives welcome message and talks to the AI immediately |
| Approval | Manual by the owner | Automatic |
| AI context | Personal assistant | Practice receptionist with configured name, hours, and identity |

### Receptionist preset

To get the most out of Business Mode, select the **Therapy Receptionist** agent in **Settings → Channels → WhatsApp → Agent**. This preset is specialized in:

- Verifying Google Calendar authentication before any action
- Checking availability and offering time slots to the client
- Creating appointments only after explicit client confirmation
- Never providing clinical or psychological guidance
- Responding in the client's language

The AI also receives the current date/time on every message, so relative references like *"tomorrow at 10am"*, *"next Monday"*, or *"in two days"* are resolved without asking the client to confirm year, weekday, or month.

<Aside type="tip">
  The receptionist preset uses Google Calendar tools (`google_list_events`, `google_create_event`). Set up the Google integration in **Settings → Integrations → Google** before enabling automatic scheduling.
</Aside>

### Appointment Calendar

To prevent the AI receptionist from reading (or accidentally writing into) your personal Google Calendar, ThairaAI uses a **dedicated calendar** for all client bookings.

Open **Settings → Integrations → Google** (visible once your Google account is connected) and use the **Appointment Calendar** panel:

| Action | When to use |
|---|---|
| **Create dedicated calendar** | First-time setup — creates a new calendar named `ThairaAI — Agendamentos (<your name>)` |
| **or choose existing…** | If you already have a calendar you use for client appointments (or one created on a previous device), pick it from the dropdown |
| **Clear** | Unlinks the calendar — the AI returns to "scheduling temporarily unavailable" mode |

The dropdown only lists calendars you have **write access** to. ThairaAI-named calendars are bubbled to the top.

#### What the AI can and cannot see

| Configuration | AI can list events | AI can create events | AI can see personal calendar |
|---|---|---|---|
| **No calendar configured** | No | Yes, on `primary` (legacy) | Limited — write only |
| **Dedicated calendar configured** | Yes, only the dedicated one | Yes, only the dedicated one | **No** |

When a dedicated calendar is set, the system prompt explicitly forbids the AI from touching the primary calendar, Gmail, or contacts. When clients ask about availability, the AI describes occupied slots in general terms ("this time isn't available") and never reveals other clients' booking titles, names, or attendees.

**Your own queries work too.** When you message your own ThairaAI assistant ("what appointments do I have today?", "list my bookings for next week"), the AI defaults to the dedicated appointment calendar — so you see exactly what the receptionist has booked, without the LLM having to read your personal Google Calendar. You can still ask it to look at your personal calendar by saying "check my personal calendar" — the underlying tools accept an explicit override.

<Aside type="caution">
  If you delete the dedicated calendar directly in Google Calendar, ThairaAI still has the ID stored. Click **Clear** in the settings panel before creating a new one, or use the **or choose existing…** dropdown to re-link.
</Aside>
