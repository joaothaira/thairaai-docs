This page covers the internal design of the native Gmail and Calendar views, the "analyze email in chat" flow, and the Gmail marker in the chat-history sidebar.

## Why native API views, not a webview

The first attempt embedded `mail.google.com` / `calendar.google.com` via an Electron `<webview>`, mirroring the WhatsApp Web tab. Google blocks account sign-in inside embedded webviews (`disallowed_useragent` — "this browser or app may not be secure"), regardless of the rendering engine or user-agent override. WhatsApp Web works only because it authenticates by QR code, never a Google account login.

The shipped design drops the webview and renders Gmail/Calendar natively, calling the Google APIs through the **existing** OAuth connection from **Settings → Google** (`GoogleIntegrationService`). The OAuth scopes already cover `gmail.readonly`, `gmail.send`, and `calendar`, so no re-consent is needed.

## Data flow

```
Renderer page (pages/google)
  → ipcBridge.googleIntegration.{status, listEmails, getEmail, listEvents}
  → googleIntegrationBridge (process)
  → GoogleIntegrationService
      → getAuthenticatedClient()  # OAuth2 client from google-integration-tokens.json
      → google.gmail(...) / google.calendar(...)
  → Google APIs
```

The renderer process holds no tokens and makes no network calls — everything goes through the IPC bridge to the main process, which owns the OAuth client.

## Design decisions

### Full email read & HTML safety

`getEmailDetail` returns the decoded body plus an `isHtml` flag. `GmailMessagePanel` renders HTML bodies inside `<iframe sandbox='' srcDoc={body}>`. The empty `sandbox` attribute disables scripts, same-origin access, form submission, and popups — the standard safe way to display untrusted email HTML without a DOMPurify dependency. Plain-text bodies render in a `<pre>` block.

### Email pagination

`listEmails` accepts a `pageToken` and returns `nextPageToken` from the Gmail `messages.list` response. `GmailView` keeps accumulated `messages` in local state and appends the next page on **Load more**; refresh resets to the first page.

### Calendar window & configured calendar

`CalendarView.fetchWindow()` derives `timeMin`/`timeMax` from the active mode and anchor date (week = the Sun–Sat of the anchor; month = the visible 6-week grid). The renderer does not pass a `calendarId`; the bridge resolves the configured `appointmentCalendarId` so the views respect the calendar chosen in **Settings → Google** instead of always querying `primary`.

### Email → chat

The "analyze in chat" action fetches the full email, builds a prompt with `buildEmailAnalysisPrompt` (HTML stripped to text), and navigates to the new-chat page:

```ts
navigate('/guid', { state: { initialInput: prompt, origin: 'gmail' } });
```

`GuidPage` seeds its draft input from `location.state.initialInput` (the user reviews, picks an assistant, and sends — reusing the entire conversation-creation pipeline rather than hardcoding a backend). `useGuidSend` reads `location.state.origin` and stamps `extra.origin = 'gmail'` on the created conversation in every backend branch.

### Gmail marker in the sidebar

`ConversationRow.renderLeadingIcon` checks `(conversation.extra as { origin?: string })?.origin === 'gmail'` and renders the Gmail logo image — mirroring the existing `conversation.name?.startsWith('wa-')` WhatsApp branch. A marker on `extra` is used instead of a name prefix because the conversation name is rendered raw in the sidebar, the chat header, and tooltips; a prefix would pollute the visible title everywhere, whereas `extra` persists in the DB and survives any retitling.
