This page covers the internal design of the WhatsApp integration for contributors.

## Message flow

### Existing contact — text message

```
User sends WhatsApp message
  → whatsapp-api stores in Postgres, fires POST /channels/whatsapp/webhook
  → whatsappWebhookHandler (Express) responds 200 immediately
  → WhatsAppPlugin.handleInboundWebhook()
      → WhatsAppAdapter.toUnifiedIncomingMessage() → IUnifiedIncomingMessage
      → (media download + normalization — see below)
      → messageHandler() (set by ChannelManager)
  → ActionExecutor.handleIncomingMessage()
      → pairing check (PairingService)
      → ActionExecutor.handleChatMessage()
          → buildBusinessContextForMessage()       # reads Business Mode config
          → sendMessage("⏳ Thinking...")           # placeholder sent to WhatsApp
          → ChannelMessageService.sendMessage()
              → business context + history injection if task is fresh
              → workerTaskManager.getOrBuildTask()  # starts Claude Code (ACP)
              → ACP stream callback per chunk
                  → editMessage() → WhatsApp drops streaming chunks
          → final editMessage({ replyMarkup: __aionuiFinal })
              → WhatsAppPlugin.editMessage()        # replyMarkup truthy → sends
              → WhatsAppPlugin.sendMessage()        # HTTP POST to whatsapp-api
  → whatsapp-api delivers reply to user
```

### Existing contact — image message

```
User sends image (with optional caption)
  → WhatsAppAdapter.toUnifiedIncomingMessage()
      → contentType = 'photo'
      → attachments = [{ type: 'photo', fileId: media.path }]
  → WhatsAppPlugin.handleInboundWebhook()
      → downloadMediaToTemp('/chat/public/image/download?q=...')
          → GET serverUrl+path with JWT auth
          → saves to $TEMP/wa-media-{timestamp}.jpg
      → attachments[0].fileName = localPath
      → normalizeMediaContent() → { type: 'text', text: '[Image: /tmp/wa-media-...jpg]\nCaption: ...' }
  → ActionExecutor → handleChatMessage() with normalized text
  → AI receives image path + caption as a text message
```

### Existing contact — voice message

```
User sends voice message (OGG/Opus)
  → WhatsAppAdapter.toUnifiedIncomingMessage()
      → contentType = 'voice'
      → attachments = [{ type: 'voice', fileId: media.path }]
  → WhatsAppPlugin.handleInboundWebhook()
      → downloadMediaToTemp('/chat/public/audio/download?q=...')
          → saves to $TEMP/wa-media-{timestamp}.ogg
      → transcribeAudio(localPath)
          → convertToWav()  # ffmpeg: OGG/Opus → WAV 16 kHz mono
          → SpeechToTextService.transcribe()  # local whisper or cloud provider
          → deletes .wav temp file
          → returns transcript text
      → if transcript: unified.content = { type: 'text', text: transcript }
      → if transcription fails: sendMessage(sttErrorNotice(err)) → return
  → ActionExecutor → handleChatMessage() with transcript
  → AI receives the spoken words as a text message
```

### New contact — Business Mode ON

```
Unknown client sends message
  → ActionExecutor.handleIncomingMessage()
      → PairingService.autoApproveUser()           # creates assistant_users record without a code
      → WhatsAppPlugin.sendMessage(welcome)        # configured welcome message
      → [falls through to normal chat flow above]
```

### New contact — Business Mode OFF

```
Unknown user sends message
  → ActionExecutor.handleIncomingMessage()
      → handlePairingShow()                        # sends 6-digit pairing code via WhatsApp
      → waits for manual approval in the panel
```

### Existing authorized contact — Business Mode paused for this chat

```
Authorized client sends message while chat is paused by the professional
  → ActionExecutor.handleIncomingMessage()
      → PairingService.isUserAuthorized() → true
      → ProcessConfig.get('assistant.whatsapp.businessMode')
          → bm.excludedChats.includes(chatId) → true
          → return  # dropped silently — professional replies manually
```

## Design decisions

### Singleton WhatsApp Web webview

The WhatsApp Web tab embeds `web.whatsapp.com` via an Electron `<webview>`. Naively mounting a `<webview>` inside `WhatsAppConversationPanel` would destroy and recreate it every time the user switches conversations — requiring a QR code re-scan.

**Solution:** `WhatsAppWebProvider` (mounted once in `AppProviders`) owns the single `<webview>` and renders it as a `position: fixed` overlay. Conversation panels expose a "slot" `<div>` via `registerSlot(el)`. The provider tracks the slot's `getBoundingClientRect()` using a `ResizeObserver` and repositions the fixed overlay to match it frame-by-frame.

Key implementation details:

- **`visibility: hidden` not `display: none`** — The slot div must always be `position: absolute; inset: 0` with `visibility: hidden/visible`. Using `display: none` collapses the element, making `getBoundingClientRect()` return zeros and causing the webview to teleport off-screen.
- **No `key` on `WhatsAppConversationPanel`** — Without a `key` prop, React reuses the panel instance across WhatsApp conversation switches. This keeps the slot mounted and the webview visible between chats.
- **`partition='persist:whatsapp-web'`** — Electron's session storage for the webview is isolated under this partition key and survives app restarts. The user scans the QR code once.
- **Chrome user agent** — WhatsApp Web detects non-Chrome browsers and refuses to load. The webview is given a Chrome 120 user agent string via the `userAgent` prop on `WebviewHost`.
- **Border radius** — `getComputedStyle(slotEl).borderRadius` is read at measurement time and applied to the fixed container with `overflow: hidden`, so the webview clips to the panel's rounded corners.

### Auto-starting the HTTP server

ThairaAI's Express server only starts when the user enables the Desktop WebUI feature. Without it, whatsapp-api webhook POSTs would get connection refused.

`WhatsAppPlugin.onStart()` calls `ensureWebServerRunning()` before registering the webhook. This auto-starts the HTTP server through the same code path as the WebUI feature — no user action required:

```ts
private async ensureWebServerRunning(): Promise<void> {
  if (getWebServerInstance()) return;
  const instance = await startWebServerWithInstance(SERVER_CONFIG.DEFAULT_PORT, false);
  setWebServerInstance(instance);
}
```

### JWT and instance lifecycle

whatsapp-api returns a JWT when creating an instance. If the instance already exists, the API returns HTTP 400/403. `fetchOrCreateJwt()` handles both cases: on conflict it falls back to `GET /instance/fetchInstances` to retrieve the existing JWT. ThairaAI can restart freely without losing the whatsapp-api session.

### `editMessage` sends instead of editing

All channel plugins implement `editMessage` for platforms that support in-place message updates (e.g., Telegram). WhatsApp has no edit API.

`WhatsAppPlugin.editMessage` ignores all calls unless `message.replyMarkup` is set. `replyMarkup` is the "stream complete" signal set only on the final message by `ActionExecutor`. Intermediate streaming chunks are silently dropped; only the final AI reply is delivered.

For this to fire, `ActionExecutor` must set `replyMarkup` on the final message. The original code only did this for WecomPlugin — `'whatsapp'` was added to the same guard:

```ts
const finalReplyMarkup =
  responseMarkup ??
  (context.platform === 'wecom' || context.platform === 'whatsapp'
    ? ({ __aionuiFinal: true } as unknown)
    : undefined);
```

### Suppressing intermediate status messages

The streaming engine calls `context.sendMessage` (not `editMessage`) for `agent_status` events like `⏳ claude` that signal which AI backend started. These would appear as separate WhatsApp messages.

`WhatsAppPlugin` tracks a per-chat set (`thinkingChats`) to know when `⏳ Thinking...` was sent but the final reply has not yet arrived. Any additional `⏳`-prefixed message during that window is suppressed:

```ts
if (isStatusMsg && this.thinkingChats.has(chatId)) {
  return `whatsapp-suppressed-${Date.now()}`;
}
```

### Image support

`WhatsAppAdapter` detects `imageMessage` and creates an attachment with `fileId` set to the whatsapp-api download URL (`/chat/public/image/download?q=...`).

`WhatsAppPlugin.handleInboundWebhook` downloads the image to a temp `.jpg` file and writes the local path into `attachments[0].fileName`. `normalizeMediaContent()` then converts the content to a plain text message:

```
[Image: /tmp/wa-media-1234567890.jpg]
Caption: what is in this photo?
```

This text is forwarded to the AI, which can read the file using its tools.

### Voice transcription

WhatsApp voice messages arrive as OGG/Opus, which whisper.cpp cannot decode directly. The pipeline converts to WAV before transcription:

```ts
// 1. Convert OGG/Opus → WAV 16 kHz mono
const wavPath = await this.convertToWav(oggPath);

// 2. Transcribe via SpeechToTextService (any configured provider)
const result = await SpeechToTextService.transcribe({
  audioBuffer: Array.from(fs.readFileSync(wavPath)),
  fileName: 'audio.wav',
  mimeType: 'audio/wav',
});

// 3. Clean up WAV; forward transcript to AI
```

`convertToWav` spawns `ffmpeg -ar 16000 -ac 1 -f wav`. The original `.ogg` is kept (it was already written to temp by `downloadMediaToTemp`); the `.wav` is deleted after transcription regardless of success or failure.

If transcription fails for any reason, the plugin replies directly to the user with a specific error notice instead of forwarding to the AI:

| Error code | Message sent |
|---|---|
| `STT_DISABLED` | "Speech-to-Text is disabled. Enable it in Settings → Tools." |
| `STT_OPENAI_NOT_CONFIGURED` | "OpenAI Whisper API key not configured…" |
| `STT_DEEPGRAM_NOT_CONFIGURED` | "Deepgram API key not configured…" |
| `STT_REQUEST_FAILED:…` | "Transcription failed: …" (includes server error detail) |
| other | "Could not transcribe voice message: …" |

### Local Whisper provider

A new `'local'` provider was added to `SpeechToTextService`. It calls a whisper.cpp-compatible HTTP server:

- **Endpoint:** `POST {url}/inference` (multipart form with `file` + optional `language`)
- **Response:** `{ text: "..." }`
- **Default URL:** `http://localhost:8080`

`LocalSpeechToTextConfig`:

```ts
type LocalSpeechToTextConfig = {
  url?: string;      // default: http://localhost:8080
  language?: string; // ISO 639-1 code, e.g. "pt"
};
```

Configure in **Settings → Tools → Speech to Text → Local Whisper**.

The whisper.cpp build lives at `whisper.cpp/` with the `ggml-base` model. The server binary is symlinked at `/Volumes/server-ssd/bin/whisper-server` (which is in `$PATH` via `~/.zshrc`).

### `lastMessageContent` type guard

The `agent_status` event can arrive *after* the text response in the ACP stream. `ActionExecutor` was tracking every stream event as the "last message content", so the final `editMessage` call could receive `{ text: '⏳ claude' }` instead of the real AI answer — which was then suppressed, resulting in no delivery.

Fixed by only updating `lastMessageContent` for `message.type === 'text'`:

```ts
if (message.type === 'text') {
  lastMessageContent = streamOutgoing;
}
```

### Conversation history injection

ThairaAI's AI sessions live in memory. After a restart or a 5-minute idle timeout, the in-memory ACP task is evicted. The next message would start a fresh Claude Code process with no context.

`ChannelMessageService.sendMessage` detects a fresh task (`workerTaskManager.getTask()` returning undefined before `getOrBuildTask` is called) and queries ThairaAI's own SQLite `messages` table for the last 30 stored turns. These are prepended to the user's message as a context block:

```
[Previous conversation:
User: Hello, how are you?
Assistant: All good! What do you need?
]

<current message>
```

Only `type === 'text'` messages are included; `⏳` placeholder messages are filtered out. History is sourced from ThairaAI's internal SQLite DB — no dependency on whatsapp-api's Postgres.

### `yoloMode` for WhatsApp

`'whatsapp'` was missing from the `isFromChannel` check in `ChannelMessageService`, causing Claude Code to wait for manual UI permission approval instead of auto-approving. Added alongside Telegram, Lark, DingTalk, WeChat, and WeCom.

### Concurrent startup scripts

Two `concurrently`-based scripts are available for development:

```bash
# App + whatsapp-api
bun run start:wa

# App + whatsapp-api + whisper server
bun run start:wa:whisper
```

`start:wa:whisper` runs whisper-server with the bundled base model:

```
whisper-server -m ../whisper.cpp/models/ggml-base.bin
```

All three processes share a single terminal with color-coded prefixes (`app`, `wa-api`, `whisper`). `--kill-others-on-fail` ensures all processes exit if any one of them crashes.

## Business Mode

Business Mode flips the paradigm: instead of the owner chatting with their own assistant, **external clients** message the professional's WhatsApp number and the AI responds on their behalf.

### Configuration

Stored in `ConfigStorage` under the `assistant.whatsapp.businessMode` key:

```ts
{
  autoApprove: boolean;      // enables the mode
  businessName?: string;     // e.g. "Silva Psychology Clinic"
  professionalName?: string; // e.g. "Dr. Ana Silva"
  businessHours?: string;    // e.g. "Mon–Fri, 9am–6pm"
  welcomeMessage?: string;   // custom first-contact message
  excludedChats?: string[];  // chatIds where the AI is paused (manual reply mode)
}
```

Config is read directly from `ProcessConfig` in the main process — no additional IPC needed.

### Auto-approval

`PairingService.autoApproveUser()` creates the `assistant_users` record without requiring a pairing code. If the user already exists it returns the existing record (idempotent). After creation it emits `channelBridge.userAuthorized` to update the panel.

### Business context injection

`ActionExecutor.buildBusinessContextForMessage()` builds a context string with the practice name, professional name, and working hours. This is passed to `ChannelMessageService.sendMessage()` and prepended to the message only when the ACP task is fresh (same pattern as history injection):

```
[System context:
You are the AI receptionist for the following professional.
Business: Silva Psychology Clinic
Professional: Dr. Ana Silva
Working hours: Mon–Fri, 9am–6pm
]

<client message>
```

### Owner notification

When Business Mode is toggled via the quick-action pill on the home screen, the app sends a message to the **owner's own WhatsApp number** via the `channel.notify-business-mode` IPC:

1. `channelBridge` retrieves the active WhatsApp plugin
2. `WhatsAppPlugin.getOwnerPhone()` calls `GET /instance/fetchInstances` and extracts `ownerJid`
3. Sends the status message directly to that number

### Per-chat exclusion (pausing the AI for one contact)

`excludedChats` is a list of WhatsApp JIDs (`5511999@s.whatsapp.net`) where the AI should stay silent and let the professional reply manually.

The exclusion is checked at two points in `ActionExecutor.handleIncomingMessage`:

1. **New contact path** — `getBusinessModeConfig(platform, chatId)` returns `null` if `chatId` is in `excludedChats`, so auto-approval is skipped and the pairing flow is shown instead.
2. **Authorized user path** — after the authorization check passes, a dedicated guard reads the raw config and returns early (drops the message silently) if the chatId is excluded:

```ts
if (isAuthorized && platform === 'whatsapp' && chatId) {
  const bm = await ProcessConfig.get('assistant.whatsapp.businessMode');
  if (bm?.excludedChats?.includes(chatId)) return;
}
```

Three ways to toggle the exclusion:

| Method | Who | Where |
|---|---|---|
| **UI toggle** | Professional | Conversation header (per-chat Switch rendered by `WhatsAppAgentToggle`) |
| **Settings list** | Admin | Settings → Channels → WhatsApp → authorized users list (per-user Switch) |
| **In-chat command** | Client or professional in the WhatsApp chat | `/pause` to add, `/resume` to remove |

### Receptionist preset

`src/process/resources/assistant/therapy-receptionist/therapy-receptionist.md` defines an assistant specialized in:
- Verifying Google Calendar authentication before any action
- Listing events, identifying free slots, and offering options to the client
- Creating appointments only after explicit client confirmation
- Never providing clinical or psychological guidance
- Responding in the client's language

## WhatsApp Skill

The WhatsApp Skill lets the **desktop agent** (main ThairaAI window) send messages and fetch recent conversation history from any WhatsApp contact on request. It uses the same protocol-block + middleware pattern as the `cron` and `weixin-file-send` skills.

### Protocol blocks

The AI emits one or more blocks inside its response text:

```
[WA_SEND]{"number":"5511999999999","text":"I'll be late"}[/WA_SEND]
[WA_FETCH]{"number":"5511999999999","limit":10}[/WA_FETCH]
```

- **`[WA_SEND]`** — fire-and-forget. Processed after the AI turn finishes. Emits a `✅ Sent to {number}` system message.
- **`[WA_FETCH]`** — two-phase. The runtime fetches recent messages from the contact, injects them as a system message, and the AI continues with that context (same pattern as `[CRON_LIST]`).

### Skill activation

`src/process/resources/skills/whatsapp/SKILL.md` has a frontmatter trigger:

```yaml
---
name: whatsapp
description: |
  Use when the user wants to send a WhatsApp message or read recent
  WhatsApp messages from a contact. Requires WhatsApp to be connected.
---
```

The file is loaded automatically when the user asks about WhatsApp messaging. It contains syntax, field descriptions, and worked examples for both blocks.

### Execution pipeline

After each AI turn, all three agent managers (`AcpAgentManager`, `GeminiAgentManager`, `AionrsManager`) call `processWaInMessage` if `hasWaCommands` is true:

```
AI response text
  → hasWaCommands() → true
  → detectWaCommands() → [{ kind: 'send', number, text }, ...]
  → processWaInMessage()
      for each command:
        send  → getActiveWhatsAppPlugin().sendMessage(jid, { type:'text', text })
                emits "✅ Sent to {number}"
        fetch → plugin.fetchMessages(jid, limit)
                formats messages as readable list
                emits result as system message → AI continues
        no plugin → emits "⚠️ WhatsApp is not connected."
```

### `fetchMessages`

`WhatsAppPlugin.fetchMessages(remoteJid, limit)` normalizes bare numbers to full JIDs (`5511999@s.whatsapp.net`) before querying:

```
GET {serverUrl}/chat/findMessages/{instanceName}?remoteJid={jid}&count={limit}
Authorization: Bearer {jwtToken}
```

Returns `WaFetchedMessage[]`:

```ts
type WaFetchedMessage = { from: string; text: unknown; timestamp: number };
```
