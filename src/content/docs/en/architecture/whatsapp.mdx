This page covers the internal design of the WhatsApp integration for contributors.

## Message flow

### Existing contact — text message

```
User sends WhatsApp message
  → whatsapp-api stores in Postgres, fires POST /channels/whatsapp/webhook
  → whatsappWebhookHandler (Express) responds 200 immediately
  → WhatsAppPlugin.handleInboundWebhook()
      → WhatsAppAdapter.toUnifiedIncomingMessage() → IUnifiedIncomingMessage
      → (media download + normalization — see below)
      → messageHandler() (set by ChannelManager)
  → ActionExecutor.handleIncomingMessage()
      → pairing check (PairingService)
      → ActionExecutor.handleChatMessage()
          → buildBusinessContextForMessage()       # reads Business Mode config
          → sendMessage("⏳ Thinking...")           # placeholder sent to WhatsApp
          → ChannelMessageService.sendMessage()
              → business context + history injection if task is fresh
              → workerTaskManager.getOrBuildTask()  # starts Claude Code (ACP)
              → ACP stream callback per chunk
                  → editMessage() → WhatsApp drops streaming chunks
          → final editMessage({ replyMarkup: __aionuiFinal })
              → WhatsAppPlugin.editMessage()        # replyMarkup truthy → sends
              → WhatsAppPlugin.sendMessage()        # HTTP POST to whatsapp-api
  → whatsapp-api delivers reply to user
```

### Existing contact — image message

```
User sends image (with optional caption)
  → WhatsAppAdapter.toUnifiedIncomingMessage()
      → contentType = 'photo'
      → attachments = [{ type: 'photo', fileId: media.path }]
  → WhatsAppPlugin.handleInboundWebhook()
      → downloadMediaToTemp('/chat/public/image/download?q=...')
          → GET serverUrl+path with JWT auth
          → saves to $TEMP/wa-media-{timestamp}.jpg
      → attachments[0].fileName = localPath
      → normalizeMediaContent() → { type: 'text', text: '[Image: /tmp/wa-media-...jpg]\nCaption: ...' }
  → ActionExecutor → handleChatMessage() with normalized text
  → AI receives image path + caption as a text message
```

### Existing contact — voice message

```
User sends voice message (OGG/Opus)
  → WhatsAppAdapter.toUnifiedIncomingMessage()
      → contentType = 'voice'
      → attachments = [{ type: 'voice', fileId: media.path }]
  → WhatsAppPlugin.handleInboundWebhook()
      → downloadMediaToTemp('/chat/public/audio/download?q=...')
          → saves to $TEMP/wa-media-{timestamp}.ogg
      → transcribeAudio(localPath)
          → convertToWav()  # ffmpeg: OGG/Opus → WAV 16 kHz mono
          → SpeechToTextService.transcribe()  # local whisper or cloud provider
          → deletes .wav temp file
          → returns transcript text
      → if transcript: unified.content = { type: 'text', text: transcript }
      → if transcription fails: sendMessage(sttErrorNotice(err)) → return
  → ActionExecutor → handleChatMessage() with transcript
  → AI receives the spoken words as a text message
```

### New contact — Business Mode ON

```
Unknown client sends message
  → ActionExecutor.handleIncomingMessage()
      → PairingService.autoApproveUser()           # creates assistant_users record without a code
      → WhatsAppPlugin.sendMessage(welcome)        # configured welcome message
      → [falls through to normal chat flow above]
```

### New contact — Business Mode OFF

```
Unknown user sends message
  → ActionExecutor.handleIncomingMessage()
      → handlePairingShow()                        # sends 6-digit pairing code via WhatsApp
      → waits for manual approval in the panel
```

### Existing authorized contact — Business Mode paused for this chat

```
Authorized client sends message while chat is paused by the professional
  → ActionExecutor.handleIncomingMessage()
      → PairingService.isUserAuthorized() → true
      → ProcessConfig.get('assistant.whatsapp.businessMode')
          → bm.excludedChats.includes(chatId) → true
          → return  # dropped silently — professional replies manually
```

## Design decisions

### Singleton WhatsApp Web webview

The WhatsApp Web tab embeds `web.whatsapp.com` via an Electron `<webview>`. Naively mounting a `<webview>` inside `WhatsAppConversationPanel` would destroy and recreate it every time the user switches conversations — requiring a QR code re-scan.

**Solution:** `WhatsAppWebProvider` (mounted once in `AppProviders`) owns the single `<webview>` and renders it as a `position: fixed` overlay. Conversation panels expose a "slot" `<div>` via `registerSlot(el)`. The provider tracks the slot's `getBoundingClientRect()` using a `ResizeObserver` and repositions the fixed overlay to match it frame-by-frame.

Key implementation details:

- **`visibility: hidden` not `display: none`** — The slot div must always be `position: absolute; inset: 0` with `visibility: hidden/visible`. Using `display: none` collapses the element, making `getBoundingClientRect()` return zeros and causing the webview to teleport off-screen.
- **No `key` on `WhatsAppConversationPanel`** — Without a `key` prop, React reuses the panel instance across WhatsApp conversation switches. This keeps the slot mounted and the webview visible between chats.
- **`partition='persist:whatsapp-web'`** — Electron's session storage for the webview is isolated under this partition key and survives app restarts. The user scans the QR code once.
- **Chrome user agent** — WhatsApp Web detects non-Chrome browsers and refuses to load. The webview is given a Chrome 120 user agent string via the `userAgent` prop on `WebviewHost`.
- **Border radius** — `getComputedStyle(slotEl).borderRadius` is read at measurement time and applied to the fixed container with `overflow: hidden`, so the webview clips to the panel's rounded corners.

### Auto-starting the HTTP server

ThairaAI's Express server only starts when the user enables the Desktop WebUI feature. Without it, whatsapp-api webhook POSTs would get connection refused.

`WhatsAppPlugin.onStart()` calls `ensureWebServerRunning()` before registering the webhook. This auto-starts the HTTP server through the same code path as the WebUI feature — no user action required:

```ts
private async ensureWebServerRunning(): Promise<void> {
  if (getWebServerInstance()) return;
  const instance = await startWebServerWithInstance(SERVER_CONFIG.DEFAULT_PORT, false);
  setWebServerInstance(instance);
}
```

### JWT and instance lifecycle

whatsapp-api returns a JWT when creating an instance. If the instance already exists, the API returns HTTP 400/403. `fetchOrCreateJwt()` handles both cases: on conflict it falls back to `GET /instance/fetchInstances` to retrieve the existing JWT. ThairaAI can restart freely without losing the whatsapp-api session.

### `editMessage` sends instead of editing

All channel plugins implement `editMessage` for platforms that support in-place message updates (e.g., Telegram). WhatsApp has no edit API.

`WhatsAppPlugin.editMessage` ignores all calls unless `message.replyMarkup` is set. `replyMarkup` is the "stream complete" signal set only on the final message by `ActionExecutor`. Intermediate streaming chunks are silently dropped; only the final AI reply is delivered.

For this to fire, `ActionExecutor` must set `replyMarkup` on the final message. The original code only did this for WecomPlugin — `'whatsapp'` was added to the same guard:

```ts
const finalReplyMarkup =
  responseMarkup ??
  (context.platform === 'wecom' || context.platform === 'whatsapp'
    ? ({ __aionuiFinal: true } as unknown)
    : undefined);
```

### Suppressing intermediate status messages

The streaming engine calls `context.sendMessage` (not `editMessage`) for `agent_status` events like `⏳ claude` that signal which AI backend started. These would appear as separate WhatsApp messages.

`WhatsAppPlugin` tracks a per-chat set (`thinkingChats`) to know when `⏳ Thinking...` was sent but the final reply has not yet arrived. Any additional `⏳`-prefixed message during that window is suppressed:

```ts
if (isStatusMsg && this.thinkingChats.has(chatId)) {
  return `whatsapp-suppressed-${Date.now()}`;
}
```

### Image support

`WhatsAppAdapter` detects `imageMessage` and creates an attachment with `fileId` set to the whatsapp-api download URL (`/chat/public/image/download?q=...`).

`WhatsAppPlugin.handleInboundWebhook` downloads the image to a temp `.jpg` file and writes the local path into `attachments[0].fileName`. `normalizeMediaContent()` then converts the content to a plain text message:

```
[Image: /tmp/wa-media-1234567890.jpg]
Caption: what is in this photo?
```

This text is forwarded to the AI, which can read the file using its tools.

### Voice transcription

WhatsApp voice messages arrive as OGG/Opus, which whisper.cpp cannot decode directly. The pipeline converts to WAV before transcription:

```ts
// 1. Convert OGG/Opus → WAV 16 kHz mono
const wavPath = await this.convertToWav(oggPath);

// 2. Transcribe via SpeechToTextService (any configured provider)
const result = await SpeechToTextService.transcribe({
  audioBuffer: Array.from(fs.readFileSync(wavPath)),
  fileName: 'audio.wav',
  mimeType: 'audio/wav',
});

// 3. Clean up WAV; forward transcript to AI
```

`convertToWav` spawns `ffmpeg -ar 16000 -ac 1 -f wav`. The original `.ogg` is kept (it was already written to temp by `downloadMediaToTemp`); the `.wav` is deleted after transcription regardless of success or failure.

If transcription fails for any reason, the plugin replies directly to the user with a specific error notice instead of forwarding to the AI:

| Error code | Message sent |
|---|---|
| `STT_DISABLED` | "Speech-to-Text is disabled. Enable it in Settings → Tools." |
| `STT_OPENAI_NOT_CONFIGURED` | "OpenAI Whisper API key not configured…" |
| `STT_DEEPGRAM_NOT_CONFIGURED` | "Deepgram API key not configured…" |
| `STT_REQUEST_FAILED:…` | "Transcription failed: …" (includes server error detail) |
| other | "Could not transcribe voice message: …" |

### Local Whisper provider

A new `'local'` provider was added to `SpeechToTextService`. It calls a whisper.cpp-compatible HTTP server:

- **Endpoint:** `POST {url}/inference` (multipart form with `file` + optional `language`)
- **Response:** `{ text: "..." }`
- **Default URL:** `http://localhost:8080`

`LocalSpeechToTextConfig`:

```ts
type LocalSpeechToTextConfig = {
  url?: string;      // default: http://localhost:8080
  language?: string; // ISO 639-1 code, e.g. "pt"
};
```

Configure in **Settings → Tools → Speech to Text → Local Whisper**.

The whisper.cpp build lives at `whisper.cpp/` with the `ggml-base` model. The server binary is symlinked at `/Volumes/server-ssd/bin/whisper-server` (which is in `$PATH` via `~/.zshrc`).

### `lastMessageContent` type guard

The `agent_status` event can arrive *after* the text response in the ACP stream. `ActionExecutor` was tracking every stream event as the "last message content", so the final `editMessage` call could receive `{ text: '⏳ claude' }` instead of the real AI answer — which was then suppressed, resulting in no delivery.

Fixed by only updating `lastMessageContent` for `message.type === 'text'`:

```ts
if (message.type === 'text') {
  lastMessageContent = streamOutgoing;
}
```

### Conversation history injection

ThairaAI's AI sessions live in memory. After a restart or a 5-minute idle timeout, the in-memory ACP task is evicted. The next message would start a fresh Claude Code process with no context.

`ChannelMessageService.sendMessage` detects a fresh task (`workerTaskManager.getTask()` returning undefined before `getOrBuildTask` is called) and queries ThairaAI's own SQLite `messages` table for the last 30 stored turns. These are prepended to the user's message as a context block:

```
[Previous conversation:
User: Hello, how are you?
Assistant: All good! What do you need?
]

<current message>
```

Only `type === 'text'` messages are included; `⏳` placeholder messages are filtered out. History is sourced from ThairaAI's internal SQLite DB — no dependency on whatsapp-api's Postgres.

### `yoloMode` for WhatsApp

`'whatsapp'` was missing from the `isFromChannel` check in `ChannelMessageService`, causing Claude Code to wait for manual UI permission approval instead of auto-approving. Added alongside Telegram, Lark, DingTalk, WeChat, and WeCom.

### Concurrent startup scripts

Two `concurrently`-based scripts are available for development:

```bash
# App + whatsapp-api
bun run start:wa

# App + whatsapp-api + whisper server
bun run start:wa:whisper
```

`start:wa:whisper` runs whisper-server with the bundled base model:

```
whisper-server -m ../whisper.cpp/models/ggml-base.bin
```

All three processes share a single terminal with color-coded prefixes (`app`, `wa-api`, `whisper`). `--kill-others-on-fail` ensures all processes exit if any one of them crashes.

## Business Mode

Business Mode flips the paradigm: instead of the owner chatting with their own assistant, **external clients** message the professional's WhatsApp number and the AI responds on their behalf.

### Configuration

Stored in `ConfigStorage` under the `assistant.whatsapp.businessMode` key:

```ts
{
  autoApprove: boolean;      // enables the mode
  businessName?: string;     // e.g. "Silva Psychology Clinic"
  professionalName?: string; // e.g. "Dr. Ana Silva"
  businessHours?: string;    // e.g. "Mon–Fri, 9am–6pm"
  welcomeMessage?: string;   // custom first-contact message
  excludedChats?: string[];  // chatIds where the AI is paused (manual reply mode)
}
```

Config is read directly from `ProcessConfig` in the main process — no additional IPC needed.

### Auto-approval

`PairingService.autoApproveUser()` creates the `assistant_users` record without requiring a pairing code. If the user already exists it returns the existing record (idempotent). After creation it emits `channelBridge.userAuthorized` to update the panel.

Auto-approved users are stamped with **`role: 'client'`**. Users who pair manually via a pairing code are stamped with **`role: 'owner'`**. This distinction drives the access isolation described below.

**Role backfill:** Users created before the `role` column was added have `role = NULL` in the database. `autoApproveUser()` detects this on the next incoming message and updates the record to `'client'` via `db.updateChannelUserRole()`. This is done lazily to avoid touching unrelated users, and ensures pre-migration business clients are correctly isolated going forward.

### Business context injection

`ActionExecutor.buildBusinessContextForMessage()` builds a context string with the practice name, professional name, working hours, **the current date/time**, and a workflow that branches on whether a dedicated appointment calendar is configured. The context is passed to `ChannelMessageService.sendMessage()` and **prepended on every message** (not just fresh tasks), so the AI never loses its receptionist persona between turns on resumed sessions:

```
[System context:
ROLE: You are the WhatsApp receptionist for Dr. Ana Silva at Silva Psychology Clinic.
Business: Silva Psychology Clinic
Professional: Dr. Ana Silva
Working hours: Mon–Fri, 9am–6pm

CURRENT DATE & TIME: 2026-05-26T23:41:54.000Z (segunda-feira, 26 de maio de 2026 às 20:41,
timezone America/Sao_Paulo).
Use this as the reference for any relative date the client mentions ("amanhã", "próxima
segunda", "daqui a 2 dias"). NEVER ask the client to confirm the year, weekday, or month.

CRITICAL — WHO YOU ARE TALKING TO:
- You are NOT talking to Dr. Ana Silva. You are talking to a CLIENT (third party).
- The Google Calendar belongs to Dr. Ana Silva, NOT to the client. NEVER say "your
  calendar", "your account", "your agenda" to the client.
- Refer to the professional in the third person; refer to the client as "você".

CALENDAR SCHEDULING WORKFLOW (with dedicated calendar configured):
1. Call google_auth_status ONCE per turn (silent check).
2. If the client asks about availability, call google_list_events on the dedicated
   appointment calendar — never the primary calendar.
3. Collect details, then call google_create_event. The event title MUST include the
   client name/phone (e.g. "Consulta — João, 553197388888").
4. Confirm in plain language; do NOT echo event IDs or technical output.

RESTRICTIONS (calendar configured):
- Read AND write only the dedicated calendar; never touch the primary calendar/Gmail/contacts.
- When listing events, only describe occupied slots in general terms ("esse horário não
  está disponível") — never reveal other clients' booking titles or attendees.
- Never modify or delete existing events.
]

<client message>
```

When the dedicated appointment calendar is **not** configured, the prompt instead instructs the AI to skip `google_list_events` entirely (since reading the primary calendar would leak private agenda data).

`GeminiDirectAgent` extracts the `[System context:…]` block via regex (`/^\[System context:\n([\s\S]*?)\n\]\n\n([\s\S]*)$/`), stores the captured text as `this.systemPrompt`, and passes it as `config.systemInstruction` to `ai.models.generateContentStream()`. The clean user text (without the context wrapper) is what gets stored in conversation history. This makes the persona a formal system instruction rather than a user-turn prefix, improving adherence.

### Client access isolation

**Security issue that was fixed:** Before this fix, every auto-approved business client received `yoloMode: true` — the same flag used for the owner's full-access sessions. This meant any external client could ask the AI to read files, access emails, run shell commands, or call any MCP tool, and those requests would be auto-approved without the professional's knowledge.

**Three-layer defence applied:**

#### Layer 1 — Identity persistence (`role` column)

`IChannelUser` now carries `role?: 'owner' | 'client'`. The field is stored in `assistant_users` (migration v27: `ALTER TABLE assistant_users ADD COLUMN role TEXT`). `rowToChannelUser` defaults `null` roles to `'owner'` for backwards compatibility.

| Who creates the record | role set |
|---|---|
| `PairingService.approvePairing()` — manual pairing code flow | `'owner'` |
| `PairingService.autoApproveUser()` — Business Mode first contact | `'client'` |

#### Layer 2 — Tool isolation at task level

`ActionExecutor.handleChatMessage()` derives `isBusinessClient = context.channelUser?.role === 'client'` and passes it to `ChannelMessageService.sendMessage()`.

`ChannelMessageService` then builds the task options:

```ts
workerTaskManager.getOrBuildTask(conversationId, {
  yoloMode: isFromChannel && !isBusinessClient,  // false for clients
  disableTools: isBusinessClient,                // true for clients
});
```

`BuildConversationOptions.disableTools` propagates to every agent type:

- **ACP (Claude Code CLI):** `AcpAgent.loadBuiltinSessionMcpServers()` returns `[]` immediately when `disableTools: true`, so no MCP server (Gmail, Drive, etc.) is injected into the session. `applySessionMode(bypassPermissions)` is also skipped, so the subprocess retains its default permission mode.
- **Gemini (CLI path):** `mcpServers` is overridden to `{}` and `yoloMode` forced `false` before the worker forks.
- **Gemini (direct API path):** When `disableTools: true`, the factory creates a `GeminiDirectAgent` instead of `GeminiAgentManager`. `GeminiDirectAgent` calls the Gemini REST API directly with no CLI, no MCP, no tool scaffolding — just streaming content generation. It maintains an in-memory conversation history (capped at 40 turns) and emits events to both the IPC bridge and `channelEventBus` so messages persist to the local SQLite DB and stream to the UI in real time.

#### Layer 3 — Permission request interception

Even with no MCP servers, the Claude Code subprocess has built-in tools (file read, bash, etc.). When the AI attempts one of these, the subprocess emits an `acp_permission` event.

`AcpAgentManager.handleSignalEvent` now handles two cases before any prompt reaches the UI or WhatsApp chat:

```ts
// Case 1 — client session: auto-deny every tool request
if (this.options.disableTools && options.length > 0) {
  const denyOption =
    options.find((o) => o.kind === 'reject_once') ??
    options.find((o) => o.kind.startsWith('reject_')) ??
    options[options.length - 1];
  setTimeout(() => void this.confirm(msg_id, callId, denyOption), 50);
  return;
}

// Case 2 — owner session: auto-approve (yoloMode)
if ((this.isYoloMode(this.currentMode) || this.options.yoloMode) && options.length > 0) {
  setTimeout(() => void this.confirm(msg_id, callId, options[0]), 50);
  return;
}
```

The `|| this.options.yoloMode` fallback on the approve path ensures the owner's session auto-approves even if `currentMode` hasn't been updated to `'bypassPermissions'` yet (e.g. a race with ACP session resume).

#### Summary

| User type | `disableTools` | `yoloMode` | Tool permission result |
|---|---|---|---|
| `role: 'owner'` (manually paired) | `false` | `true` | Auto-approved |
| `role: 'client'` (auto-approved, Business Mode) | `true` | `false` | Auto-denied; AI sees rejection and responds in text |

### Per-chat exclusion (pausing the AI for one contact)

`excludedChats` is a list of WhatsApp JIDs (`5511999@s.whatsapp.net`) where the AI should stay silent and let the professional reply manually.

The exclusion is checked at two points in `ActionExecutor.handleIncomingMessage`:

1. **New contact path** — `getBusinessModeConfig(platform, chatId)` returns `null` if `chatId` is in `excludedChats`, so auto-approval is skipped and the pairing flow is shown instead.
2. **Authorized user path** — after the authorization check passes, a dedicated guard reads the raw config and returns early (drops the message silently) if the chatId is excluded:

```ts
if (isAuthorized && platform === 'whatsapp' && chatId) {
  const bm = await ProcessConfig.get('assistant.whatsapp.businessMode');
  if (bm?.excludedChats?.includes(chatId)) return;
}
```

Three ways to toggle the exclusion:

| Method | Who | Where |
|---|---|---|
| **UI toggle** | Professional | Conversation header (per-chat Switch rendered by `WhatsAppAgentToggle`) |
| **Settings list** | Admin | Settings → Channels → WhatsApp → authorized users list (per-user Switch) |
| **In-chat command** | Client or professional in the WhatsApp chat | `/pause` to add, `/resume` to remove |

### GeminiDirectAgent

`src/process/task/GeminiDirectAgent.ts` is a lightweight `IAgentManager` implementation for business-client Gemini sessions. It does **not** disable tool calls — instead it declares a narrow, hardcoded toolset and executes those calls in-process, bypassing both the Gemini CLI and the MCP subprocess.

Key properties:
- **No subprocess** — calls `@google/genai` `generateContentStream` directly in-process.
- **In-memory history** — maintains `Array<{ role, parts }>` (supporting `text`, `functionCall`, and `functionResponse` parts) capped at 40 turns to bound memory.
- **API key resolution** — uses `model.apiKey` if set; falls back to `process.env.GEMINI_API_KEY`. The explicit value is always passed to `GoogleGenAI({ apiKey })` so other agents' `prepareEnvironment()` calls that clear the env var do not affect this agent.
- **System instruction extraction** — on each `sendMessage` call, strips the `[System context:\n…\n]` prefix injected by `ChannelMessageService`, stores the extracted text as `this.systemPrompt`, and passes it as `config.systemInstruction` in the API call. The clean user text is stored in history.
- **Hardcoded function declarations** — every request includes a `tools: CALENDAR_TOOLS` config with three function declarations: `google_auth_status`, `google_create_event`, and `google_list_events`. Gemini cannot call any other tools; there is no MCP, no shell, no file read.
- **Multi-turn tool loop** — `sendMessage()` runs a `while (continueLoop && toolTurns < MAX_TOOL_TURNS)` loop. Each iteration streams a response; if any `functionCall` parts arrive they're executed via `executeTool()` (which dispatches to `googleIntegrationService.getStatus()` / `createEvent()` / `listEvents()`, the latter two using `getAppointmentCalendarId()` to read from `google.integration.appointmentCalendarId`, falling back to `'primary'`). The results are appended to history as `functionResponse` parts and the loop continues until Gemini stops calling tools. Cap is 5 turns to prevent runaway loops.
- **User-message persistence** — immediately after the system-context extraction, the agent calls `addMessage(conversation_id, { type: 'text', position: 'right', … })` so the client's incoming message appears in the chat UI alongside replies. (Earlier versions skipped this and only left-side assistant rows were visible.)
- **Tool-call visibility** — for each function call, the agent emits a `tool_group` event twice with the same `toolMsgId`: first with `status: 'Executing'` and a Portuguese description (e.g. "Verificando conexão com Google Calendar"), then with `status: 'Success'` / `'Error'` and `resultDisplay` set to the tool's return value. The existing `composeMessage` logic merges these by msg_id so the UI shows a live status flip.
- **Dual emission** — `emit()` sends each event to both `ipcBridge.geminiConversation.responseStream` (real-time UI) and `channelEventBus.emitAgentMessage` (WhatsApp reply delivery), then calls `addOrUpdateMessage` to persist to SQLite.

### Dedicated appointment calendar

Receptionist sessions need to schedule appointments without leaking the professional's personal agenda. ThairaAI persists a Google Calendar ID in `google.integration.appointmentCalendarId` (with `appointmentCalendarName` for display); all Gemini tool calls and the system prompt branch on whether it's set.

**Calendar lifecycle (`GoogleIntegrationService`):**
- `createCalendar(summary)` calls `calendar.calendars.insert` with the host's local timezone and returns `{ id, summary }`.
- `listCalendars()` calls `calendarList.list({ minAccessRole: 'writer' })`, so only calendars the user can write to appear in the picker.
- `createEvent(params, calendarId = 'primary')` and `listEvents(maxResults, calendarId = 'primary')` accept an explicit `calendarId` — defaulting to `'primary'` preserves the original behaviour for non-receptionist callers.

**Settings UI (`GoogleSettings.tsx`):**
The Appointment Calendar panel renders only when `status.connected` is true. When no calendar is configured it shows two controls side-by-side:
- **"Create dedicated calendar"** — primary button. Default name is `ThairaAI — Agendamentos (<professionalName>)` if the WhatsApp business-mode `professionalName` field is set, otherwise just `ThairaAI — Agendamentos`. Persists immediately to `google.integration`.
- **"or choose existing…"** Select — lazy-loads writable calendars on focus (`loadAvailableCalendars()` caches the result so the IPC is only called once per session). Calendars are sorted: ThairaAI/Agendamento-named first, then `primary`, then alphabetical. Picking one persists immediately. Useful if the user has an existing dedicated calendar from another tool, or wants to dogfood with their personal calendar.

When a calendar is already configured the panel collapses to its name and a Clear button.

**Agent-side resolution (`GeminiDirectAgent.getAppointmentCalendarId()`):**

```ts
async function getAppointmentCalendarId(): Promise<string> {
  const cfg = await ProcessConfig.get('google.integration');
  const id = cfg?.appointmentCalendarId?.trim();
  return id && id.length > 0 ? id : 'primary';
}
```

Called inside `executeTool()` for both `google_create_event` and `google_list_events`. If no calendar is configured, `google_list_events` returns a string telling the model that no calendar exists rather than dumping the primary calendar's contents — defence in depth on top of the prompt-level restriction.

**Prompt-level restriction (`ActionExecutor.buildBusinessContextForMessage()`):**
The system prompt reads `google.integration.appointmentCalendarId` and branches:
- *Configured*: instructs the AI to call `google_list_events` for availability questions, only describe occupied slots in general terms, never reveal other clients' booking titles, and place all new events on the dedicated calendar.
- *Not configured*: explicitly forbids calling `google_list_events`; the AI may only call `google_auth_status` and `google_create_event`.

**Owner / ACP sessions:**
The receptionist plumbing above only fires for `role: 'client'` sessions (the WhatsApp contact who is talking *to* the receptionist). The owner — i.e. the professional themselves chatting with their own ThairaAI assistant — takes the standard ACP path with full Claude Code tooling, and reaches the calendar through the built-in `aionui-google-integration` MCP server.

That MCP server is a stdio subprocess spawned via the `mcp.config` entry persisted in `aionui-config.txt`. Its `google_list_events` and `google_create_event` tools both read `process.env.APPOINTMENT_CALENDAR_ID` (via `getDefaultCalendarId()`) and use it as the default `calendarId`, with an explicit `calendar_id` tool parameter for opt-out (e.g. `calendar_id: 'primary'` to query the professional's personal calendar). The env var is populated by `buildGoogleEnv()` in `initStorage.ts`, which reads `google.integration.appointmentCalendarId` from the config file at spawn time.

When the user creates or selects an appointment calendar in **Settings → Integrations → Google**, the renderer calls one of two process-side IPC methods (`googleIntegration.createCalendar` or `googleIntegration.setAppointmentCalendar`). Both methods:
1. Persist the new value to `google.integration` in `aionui-config.txt`
2. Call `refreshGoogleMcpEnv()`, which re-stamps the env block of the `aionui-google-integration` entry in `mcp.config`

Newly spawned MCP subprocesses pick up the new `APPOINTMENT_CALENDAR_ID` automatically. Existing in-flight Claude sessions keep their original env until their MCP subprocess restarts — typically the next message after an idle gap, or after the user explicitly restarts the agent.

This means when the Dr. messages himself ("what appointments do I have today?"), Claude calls `google_list_events`, the MCP defaults to the dedicated appointment calendar (not primary), and the Dr. sees only the bookings made by the receptionist — never accidentally exposing his personal Google Calendar to the LLM.

### Stale session validation

If the user deletes a conversation from the ThairaAI UI while a WhatsApp session is active, subsequent messages would fail with "Conversation not found" because `SessionManager` still held the old `conversationId`.

`ActionExecutor.handleIncomingMessage` now validates the session before using it:

```ts
  if (!check.success || !check.data) {
    await sessionManager.clearSession(channelUser.id, chatId);
    session = null;  // triggers fresh conversation creation below
  }
}
```

### Google Calendar — `shell.openExternal` in workers

`GoogleIntegrationService.connect()` used `electron.shell.openExternal` to launch the browser OAuth flow. When called from a Gemini CLI tool registered in a forked worker process, `electron.shell` is `undefined`, causing a crash.

Fixed by adding a `openUrl()` helper that guards the Electron API and falls back to `exec` with platform-specific commands (same pattern as the standalone `googleIntegrationServer.ts`):

```ts
function openUrl(url: string): Promise<void> {
  if (shell?.openExternal) return shell.openExternal(url).catch(() => {});
  const cmd = process.platform === 'darwin' ? `open "${url}"`
             : process.platform === 'win32'  ? `start "" "${url}"`
                                             : `xdg-open "${url}"`;
  return new Promise((resolve) => exec(cmd, () => resolve()));
}
```

## WhatsApp Skill

The WhatsApp Skill lets the **desktop agent** (main ThairaAI window) send messages and fetch recent conversation history from any WhatsApp contact on request. It uses the same protocol-block + middleware pattern as the `cron` and `weixin-file-send` skills.

### Protocol blocks

The AI emits one or more blocks inside its response text:

```
[WA_SEND]{"number":"5511999999999","text":"I'll be late"}[/WA_SEND]
[WA_FETCH]{"number":"5511999999999","limit":10}[/WA_FETCH]
```

- **`[WA_SEND]`** — fire-and-forget. Processed after the AI turn finishes. Emits a `✅ Sent to {number}` system message.
- **`[WA_FETCH]`** — two-phase. The runtime fetches recent messages from the contact, injects them as a system message, and the AI continues with that context (same pattern as `[CRON_LIST]`).

### Skill activation

`src/process/resources/skills/whatsapp/SKILL.md` has a frontmatter trigger:

```yaml
---
name: whatsapp
description: |
  Use when the user wants to send a WhatsApp message or read recent
  WhatsApp messages from a contact. Requires WhatsApp to be connected.
---
```

The file is loaded automatically when the user asks about WhatsApp messaging. It contains syntax, field descriptions, and worked examples for both blocks.

### Execution pipeline

After each AI turn, all three agent managers (`AcpAgentManager`, `GeminiAgentManager`, `AionrsManager`) call `processWaInMessage` if `hasWaCommands` is true:

```
AI response text
  → hasWaCommands() → true
  → detectWaCommands() → [{ kind: 'send', number, text }, ...]
  → processWaInMessage()
      for each command:
        send  → getActiveWhatsAppPlugin().sendMessage(jid, { type:'text', text })
                emits "✅ Sent to {number}"
        fetch → plugin.fetchMessages(jid, limit)
                formats messages as readable list
                emits result as system message → AI continues
        no plugin → emits "⚠️ WhatsApp is not connected."
```

### `fetchMessages`

`WhatsAppPlugin.fetchMessages(remoteJid, limit)` normalizes bare numbers to full JIDs (`5511999@s.whatsapp.net`) before querying:

```
GET {serverUrl}/chat/findMessages/{instanceName}?remoteJid={jid}&count={limit}
Authorization: Bearer {jwtToken}
```

Returns `WaFetchedMessage[]`:

```ts
type WaFetchedMessage = { from: string; text: unknown; timestamp: number };
```
